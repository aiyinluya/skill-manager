## 1. 为什么重构（Why）

### 1.1 v1.0 复盘
v1.0 定位为**本地桌面软件（Tauri 2）**，解决「多 AI 编码工具的 skills 散落、无法统一管理」的痛点，交付了全局扫描、浏览、安装/卸载、跨编辑器同步、一致性检查、场景管理。

它回答的是：**「我机器上有哪些 skill、怎么在编辑器间搬」**。

### 1.2 发现访谈暴露的缺口（Gap）
通过需求发现（掘需需框架，SPIN + Gap Selling）确认：

| 维度 | 现状（v1.0） | 用户的真实期望 |
|------|-------------|---------------|
| 视角 | 仅**全局**清单（184 个 skill 平铺） | 缺**项目**视角 |
| 项目概念 | 无 | 「项目 = 独立文件夹」，可多副本、跨编辑器 skills 不同 |
| skills 分层 | 只有全局 | **项目自带 + 可叠加全局** 两层 |
| 相关性 | 无信号 | 要能区分「**本项目能用** / **有但用不上**」 |
| 配置 | 仅全局同步配置 | **自动推断为主 + 用户可指定路径/覆盖** |

用户原话：「只给全局清单也需要，但**用处不够大**。」

**结论**：没有项目感知，这工具只是「好看的目录浏览器」（nice-to-have）。补上**项目感知 + 相关性分层**，它才升级为「按项目给我对的 skill 的助手」（must-have，差异化护城河）。这即是本次重构的核心。

### 1.3 重构目标
1. 在保留全局管理能力的前提下，引入**项目上下文**作为一等公民。
2. 让用户打开任意项目文件夹时，立刻看清：**哪些 skill 属于/适用于本项目，哪些虽有但本项目用不上**。
3. 支持**项目自带 skills** 与**全局 skills** 双层级聚合与区分。
4. 以**自动推断为主、用户覆盖兜底**的方式判定相关性，不增加用户负担。

## 2. 目标用户与使用场景

**核心用户**：同时使用 ≥2 个 AI 编码工具、需要统一管理 skills 的开发者（如李：Cursor / Claude Code / Codex / WorkBuddy 并存）。

**典型场景**

- **S1 打开项目想快速启用对的 skill**：开发者 clone 一个 Python/FastAPI 项目，打开工具，希望直接看到「这个项目该用哪些」，而不是在 184 个全局 skill 里自己翻。
- **S2 项目自带 skills**：某项目在 `.claude/skills` 下放了团队专属 skill，工具要能识别并标注「来源=本项目」。
- **S3 多副本一致性**：同一项目在本地复制了多份（如 `proj/` 与 `proj-backup/`），希望配置可在副本间共享或各自独立（见 B6）。
- **S4 跨编辑器统一**：同一项目用 Cursor 写前端、用 Claude Code 写后端，两边 skills 不同，工具要统一呈现并支持分别同步。
- **S5 团队协同**：把项目能力集（启用/屏蔽哪些 skill）提交进 git， teammates clone 后获得一致配置。

## 3. 核心概念定义

| 概念 | 定义 |
|------|------|
| **项目（Project）** | 一个**独立文件夹**。同一逻辑项目可本地存在多份副本（多个文件夹）；同一项目在不同编辑器下的 skills 可不同。 |
| **项目级 skills** | 位于**项目目录内**、按编辑器组织的 skills（如 `<项目>/.claude/skills`、`<.cursor/skills>`、`<.codex/skills>` 等）。随项目走，通常提交进 git。 |
| **全局 skills** | 各编辑器的**全局** skills 目录 + 中心仓库（central-skills）。与具体项目无关。 |
| **相关性（Relevance）** | 某 skill 是否适用于「当前打开的项目」。项目级 skill 恒为相关；全局 skill 按推断信号 + 用户覆盖判定。 |
| **项目配置（.skillconfig）** | 描述本项目 skill 相关性的配置文件：启用/屏蔽清单、额外路径、推断修正。详见 B6。 |
| **项目身份（projectId）** | 用于关联「同一项目的多份副本」的标识：优先取自 `.skillconfig`，否则 git remote URL，否则路径哈希。 |
| **用户标注（User Annotation）** | 用户对某 skill 的私有元数据，含**备注（notes）**与**重要性（importance）**。跟随 skill 身份（name + 来源编辑器/源）持久化，跨项目共享、可在项目内覆盖。 |
| **技能应用商店源（Marketplace Source）** | 用户可配置的「技能源」：GitHub 仓库 / 本地目录 / 远程索引 URL。工具从这些源拉取可安装的 skill 清单，形成可浏览的商店。 |

## 4. 设计原则与非目标（Non-goals）

**设计原则**
- **Why 优先**：每个功能先讲清解决什么用户问题，再列实现。
- **双层级清晰**：项目级与全局级永远可区分，不混为一谈。
- **自动为主、覆盖兜底**：相关性默认自动推断，用户可一键修正，修正可持久化。
- **只读结构、不读源码**：项目扫描只解析 SKILL.md frontmatter 与依赖清单文件名，不读取业务源代码内容。

**非目标（本次明确不做）**
- 不做 skill **付费 / 商业化 / 社区 UGC 运营**：技能应用商店（D 层）以**用户自定义源 + 本地优先**为主，不做平台账号体系、不做官方中心化市场垄断。
- 不做**跨设备云同步**（配置随项目文件夹 / 可选中心仓库，但不做云端账号体系）。
- 不做 skill **内容创作编辑器**（v1.0 的 F14 推迟，本期不纳入）。
- 不做 **LLM 相关性推理**：相关性基于确定性信号（依赖/框架/标签/已引用），可解释、可离线、零成本。

## 6. 用户故事

- 作为**多编辑器开发者**，打开项目文件夹时，我希望工具自动识别项目并只高亮「本项目该用的 skill」，以便快速启用，而不被 184 个全局技能淹没。
- 作为**项目owner**，我希望把项目专属 skills 放在项目目录下并提交 git，以便团队成员 clone 后开箱即得一致能力。
- 作为**用户**，当工具误判某 skill 相关性时，我希望一键「屏蔽/启用」并让它记住，以便推断越用越准。
- 作为**多副本使用者**，我希望 `proj/` 与 `proj-backup/` 被视为同一项目（共享配置）或各自独立（见 B6），以便符合我的工作习惯。
- 作为**团队 lead**，我希望项目能力集可随仓库分发，以便新人无需手动配置就能用对 skills。
- 作为**高频使用者**，我希望给常用 skill 加备注（如「这个用于 X 项目的部署脚本」）并标重要性，以便在列表里一眼看到重点、不被 184 个 skill 淹没。
- 作为**希望扩展能力的用户**，我希望配置自己的技能源（团队 GitHub 仓库 / 本地目录）并在商店里浏览安装，以便不必每次手敲 git 命令装 skill。

## 7. 相关性模型（B3 详细设计）

**输入信号（全部本地、确定性、可解释）**

1. **项目技术栈 / 依赖清单**：检测项目根及子目录的 `package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod`、`requirements.txt`、`pom.xml` 等，提取语言/框架。
2. **框架标记文件**：如 `next.config.js`、`django` 痕迹、`vite.config`、`tsconfig.json`、`.rails` 等。
3. **skill 自声明适用条件**：SKILL.md frontmatter 中的 `applies_to` / `tags` / `stack`（若社区逐步采用）。
4. **项目已引用的 skill**：项目里已存在的 skill 名称/触发词，作为强信号。
5. **用户覆盖**（B4）：显式启用/屏蔽，优先级最高。

**判定规则（简单、可解释，非 ML）**

| 信号命中 | 相关性分层 | UI 归属 |
|----------|-----------|---------|
| 位于项目目录下 | **本项目** | 区① 本项目 skills |
| 全局 skill，且 applies_to/tags 命中项目技术栈，或名称/触发词命中已用框架 | **推荐（强相关）** | 区② 推荐用于本项目 |
| 全局 skill，通用型或弱信号命中 | **可用（弱相关）** | 区③ 全局其他（标注「可用但本项目未匹配」） |
| 全局 skill，无任何信号命中 | **不相关** | 区③ 全局其他（折叠/降权） |
| 用户显式屏蔽 | **屏蔽** | 不展示 / 可展开查看 |

> 设计约束：推断必须**可解释**——每个「推荐」都要能溯源到具体信号（如「命中：pyproject.toml → FastAPI」），UI 上悬停可见，避免黑盒。

## 8. 项目配置存储决策（B6，待李确认）

**问题**：同项目多文件夹副本时，项目配置（启用/屏蔽/覆盖）归谁？

| 方案 | 存储位置 | 优点 | 缺点 |
|------|---------|------|------|
| **(a) 写进项目目录** | `<项目>/.skillconfig`（JSON） | 跟文件夹走；git clone / 换机器自带；与「项目级 skills 在项目目录下」自洽 | 多副本之间不自动互通 |
| **(b) 中心仓库索引** | `~/.skill-manager/profiles/<projectId>.json` | 多份副本共享一份配置（按 projectId 关联） | 配置不跟项目文件夹走；换机器需单独同步 |

**产品通推荐（默认决策，待李拍板）**
- **以 (a) 为默认**：`.skillconfig` 随项目走，契合 S2/S5 团队协同场景，且零额外基础设施。
- **以 (b) 为可选增强**：仅当「同项目多副本需共享配置」或「团队集中管控」时启用，通过 projectId 关联。
- 两者不互斥：(a) 是单人/单项目的真相源；(b) 是跨副本/跨机的同步层。

> ⚠️ 此决策影响 B1 的 projectId 解析与 B4 的写入路径。在李确认前，实现先按 (a) 落地，(b) 作为 Next 阶段开关。

## 10. 指标体系

**北极星指标**
- **项目相关 skill 周采纳数**：用户在「项目上下文」中，通过本工具启用/确认适用的 skill 的周活跃计数。（衡量工具是否真的在「帮用户用对 skill」）

**驱动指标**
| 指标 | 定义 | 基线（占位，待埋点校准） | 目标 |
|------|------|------------------------|------|
| 项目识别准确率 | 打开文件夹后正确绑定 projectId 的比例 | — | ≥ 98% |
| 相关性推荐采纳率 | 推荐区中被用户采纳 / 推荐区总数 | — | ≥ 60% |
| 双层级安装量 | 项目级 + 全局被启用总数 | — | 逐周上升 |
| .skillconfig 渗透率 | 打开过的项目中含 .skillconfig 的比例 | — | ≥ 40% |
| 备注渗透率 | 被加过备注的 skill 占用户可见 skill 比例 | — | ≥ 30% |
| 重要性标记使用率 | 被标重要性（非默认）的 skill 比例 | — | ≥ 25% |
| 商店安装量 | 从自定义源商店安装/更新的 skill 周计数 | — | 逐周上升 |

**健康指标**
| 指标 | 定义 | 警戒线 |
|------|------|--------|
| 误伤率 | 相关 skill 被误判为「不相关/用不上」的比例 | < 5% |
| 配置冲突率 | 多副本 (a)(b) 配置不一致的报错率 | < 2% |
| 同步失败率 | 沿用 A7 同步失败率 | < 1% |
| 源失效率 | 配置的商店源中不可达/解析失败的比例 | < 10% |

> 基线为占位，需接入前端埋点后用真实数据回填；指标必须可观测、可归因。

## 12. 边界、风险与合规

- **隐私**：`.skillconfig` 若含本地绝对路径，提交 git 可能泄露用户名（如 `C:\Users\liz-an\...`）。默认存**相对路径**；写入时提示「勿提交含绝对路径的敏感项」。
- **技术可行性**：各编辑器在项目内的 skills 子目录路径随版本演进 → 用**可配置映射表**（编辑器 id → 项目内子路径），不硬编码。
- **范围控制**：严格守住 §4 非目标，不在本期引入 LLM 推理 / 云账号 / 创作编辑器。
- **性能**：项目扫描在「打开文件夹」时触发，可异步 + 骨架屏；大仓库跳过 `node_modules`/`.git`（现有逻辑已处理）。

## 13. 验收标准

1. 双击 `.exe` 启动，用户机无 Node / Rust / 命令行依赖（沿用 v1.0）。
2. 用户「打开文件夹」指定项目后，工具正确识别 projectId 并聚合**项目级 + 全局**两层 skills。
3. UI 三区（本项目 / 推荐用于本项目 / 全局其他）正确渲染；每个推荐可溯源到具体信号。
4. 给定标注测试集（如 FastAPI 项目），自动推断 top 相关 skill 的**误伤率 < 5%**。
5. 用户覆盖（启用/屏蔽）持久化到 `.skillconfig`，且在副本 / git clone 后生效。
6. 危险操作（删除/移动/覆盖/写配置）前强制确认。
